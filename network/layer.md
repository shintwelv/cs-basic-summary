# OSI 7 계층 요약표

각 계층은 독립적인 역할을 수행하며, 상위 계층은 하위 계층을 도구로 사용하여 통신한다.

| 계층 | 이름 (Layer) | 주요 역할 | 주요 주소/장비 | 데이터 단위 (PDU) |
| :--- | :--- | :--- | :--- | :--- |
| **L7** | 응용 (Application) | 사용자 인터페이스, 네트워크 서비스 제공 | HTTP, DNS, FTP | Data / Message |
| **L6** | 표현 (Presentation) | 데이터 형식 변환, 암호화, 압축 | SSL/TLS, JPEG | Data |
| **L5** | 세션 (Session) | 통신 세션 유지 및 관리 (동기화) | API, Sockets | Data |
| **L4** | 전송 (Transport) | 포트 번호를 통한 프로세스 식별, 신뢰성 제어 | TCP/UDP, Port | Segment |
| **L3** | 네트워크 (Network) | IP 주소를 통한 최적 경로 결정 (라우팅) | IP, Router | Packet |
| **L2** | 데이터 링크 (Data Link) | 물리 주소를 통한 인접 장치 간 통신 | MAC, Switch | Frame |
| **L1** | 물리 (Physical) | 전기/광 신호를 전송 매체로 전달 | Hub, Cable | Bit |

## 캡슐화 & 역캡슐화 원리

데이터는 송신 측에서 아래로 내려가며 '헤더'를 입고, 수신 측에서 위로 올라가며 '헤더'를 벗는다.

- **캡슐화 (Encapsulation)**: L7(Data) → L4(Header+Data) → L3(Header+Data) → L2(Header+Data+Trailer)
  - 각 층을 지날 때마다 **운송 정보(송장)**가 붙는 과정이다.
- **역캡슐화 (Decapsulation)**: L1 → L2(Check MAC) → L3(Check IP) → L4(Check Port) → L7
  - 수신자가 자신의 주소가 맞는지 확인하며 포장을 하나씩 뜯는 과정이다.

## "브라우저에 www.google.com을 입력하면 일어나는 일"

### Step 1: 목적지 IP 주소 찾기 (DNS - L7)
- 브라우저는 먼저 www.google.com의 실제 주소(IP)를 알아내기 위해 DNS 서버에 요청을 보낸다.
- DNS는 응용 계층(L7) 프로토콜이며, UDP(L4)를 타고 전송된다.

### Step 2: 연결 수립 (TCP 3-Way Handshake - L4)
- 목적지 IP를 알아냈다면, 브라우저는 구글 서버와 신뢰할 수 있는 통로를 만든다.
- TCP 프로토콜을 사용하여 "연결해도 될까요?" 등의 3단계 확인 과정을 거친다.

### Step 3: 데이터 요청 (HTTP Request - L7)
- 연결이 되면 브라우저는 서버에 HTTP GET 메시지를 보낸다. "구글 메인 페이지 좀 보여줘!"라는 의미이다.

### Step 4: 전송 및 라우팅 (L3, L2, L1)
- 요청 메시지는 캡슐화된다.
  - **L4**: 출발지/목적지 포트(80 또는 443) 추가
  - **L3**: 출발지/목적지 IP 주소 추가
  - **L2**: 인접 기기(공유기 등)의 MAC 주소 추가
  - **L1**: 전기 신호(Bit)가 되어 광케이블을 타고 전 세계로 이동

### Step 5: 서버의 응답 및 화면 표시
- 구글 서버는 역캡슐화를 통해 요청을 읽고, 결과물(HTML, 이미지 등)을 다시 캡슐화해서 보낸다.
- 브라우저는 받은 데이터를 **6계층(표현)** 에서 복호화/압축해제하고, **7계층(응용)** 인 브라우저 화면에 최종적으로 띄워준다.
