# 메모리 관리 (Memory Management)

## 1. 가상 메모리와 주소 바인딩

> **Q:**
> 메모리를 왜 운영체제가 일일이 관리해야 하는가?

> **A:**
> 여러 프로세스가 하나의 메모리를 마구잡이로 사용하면 서로의 데이터를 침범하여 시스템이 붕괴될 수 있기 때문이다. 이를 방지하기 위해 운영체제는 **보호(Protection)** 와 **격리(Isolation)** 를 수행하여 각 프로세스가 안전하게 실행될 수 있는 환경을 보장한다.

> **Q:**
> 프로세스가 바라보는 주소와 실제 RAM의 주소는 동일한가?

> **A:**
> 아니다. 프로세스는 논리적인 가짜 주소인 **가상 주소(Virtual Address)** 를 보고, 실제 RAM은 **물리 주소(Physical Address)** 를 가진다. CPU가 가상 주소로 접근할 때 이를 실제 물리 주소로 번역해 주는 하드웨어 장치가 바로 **MMU(Memory Management Unit)** 이다.

## 2. 메모리 할당과 단편화

> **Q:**
> 메모리를 통째로 빌려줄 때 발생하는 '빈 공간 낭비' 문제는 무엇인가?

> **A:**
> 이를 **외부 단편화(External Fragmentation)** 라고 한다. 총 빈 공간의 합은 프로세스를 수용하기에 충분하지만, 조각조각 나 있어 연속된 큰 덩어리가 없기 때문에 프로세스를 할당하지 못하는 현상이다.

> **Q:**
> 이러한 단편화 문제를 해결하기 위해 도입된 '조각내기' 기법은 무엇인가?

> **A:**
> **페이징(Paging)** 기법이다. 프로세스를 고정된 크기인 **페이지(Page)** 로 자르고, 실제 메모리를 **프레임(Frame)** 이라는 같은 크기의 조각으로 나누어 빈 곳마다 조각조각 집어넣는 방식이다. 이를 통해 외부 단편화를 근본적으로 해결할 수 있다.

## 3. 페이지 관리와 성능 최적화

> **Q:**
> 페이징을 수행할 때 어떤 정보가 필요하며, 새로운 관리 이슈는 무엇인가?

> **A:**
> 페이지 조각들이 각각 어떤 물리 프레임에 있는지 기록한 **페이지 테이블(Page Table)** 이라는 장부가 필요하다. 또한, 페이지 크기가 고정되어 있어 마지막 조각이 페이지 크기보다 작을 때 발생하는 공간 낭비인 **내부 단편화(Internal Fragmentation)** 가 발생할 수 있다.

> **Q:**
> 주소 번역 속도를 높이기 위한 고속 캐시 장치는 무엇인가?

> **A:**
> **TLB(Translation Lookaside Buffer)** 이다. 자주 사용하는 주소 변환 정보를 담아두는 하드웨어 캐시로, 주소 번역 시 메인 메모리의 페이지 테이블에 접근하기 전에 먼저 확인하여 속도를 비약적으로 향상시킨다.

> **Q:**
> 메모리에 필요한 데이터가 없어서 작업을 중단하고 불러와야 하는 상황은 무엇인가?

> **A:**
> **페이지 폴트(Page Fault)** 라고 한다. 이때 운영체제는 하드디스크나 SSD에서 해당 데이터를 메모리로 가져오는 작업을 수행하며, 작업이 완료된 후 프로세스를 다시 실행한다.

> **Q:**
> 메모리가 가득 찼을 때 어떤 조각을 내보내는 것이 가장 효율적인가?

> **A:**
> 가장 오랫동안 사용되지 않은 조각을 교체 대상으로 삼는 **LRU(Least Recently Used)** 알고리즘이 현대 운영체제에서 가장 널리 쓰인다. 이는 과거의 사용 패턴을 통해 가까운 미래에 사용될 가능성이 낮은 데이터를 합리적으로 예측하는 방식이다.
