# 인터럽트와 시스템 콜 (Interrupt & System Call)

운영체제는 **인터럽트 구동 방식(Interrupt-Driven)** 으로 동작하는 거대한 이벤트 처리기입니다. CPU가 프로그램을 실행하는 도중 발생하는 내/외부의 신호를 처리하고, 사용자 프로그램이 운영체제의 서비스를 안전하게 요청하는 메커니즘을 이해하는 것이 핵심입니다.

---

## 1. 인터럽트 (Interrupt)

인터럽트는 CPU가 프로그램을 실행하고 있을 때, 외부 혹은 내부에서 발생하는 예기치 못한 사건을 알리는 신호입니다. 인터럽트가 발생하면 CPU는 즉시 실행 중이던 코드를 멈추고 제어권을 운영체제로 넘깁니다.

### A. 인터럽트의 종류

1. **하드웨어 인터럽트 (External Interrupt)**
   - **발생원**: CPU 외부의 하드웨어 장치 (마우스 클릭, 키보드 입력, 타이머 종료, 입출력 완료 등).
   - **목적**: 주변 장치가 CPU에 특정 작업의 완료나 상태 변화를 알리기 위해 사용합니다.
2. **소프트웨어 인터럽트 (Internal Interrupt / Trap)**
   - **예외(Exception)**: 프로그램 실행 도중 발생하는 오류 (0으로 나누기, 잘못된 메모리 주소 참조, 권한 없는 명령어 실행 등).
   - **시스템 콜(System Call)**: 사용자 프로그램이 커널 서비스를 요청하기 위해 의도적으로 호출하는 `trap` 명령어.

> [!QUESTION] 인터럽트와 폴링(Polling)의 차이점은 무엇인가요?
> 
> **폴링**은 CPU가 주기적으로 장치의 상태를 확인하며 대기하는 방식(Busy Waiting)인 반면, **인터럽트**는 장치가 신호를 보낼 때까지 CPU가 다른 작업을 수행할 수 있어 자원 효율성이 훨씬 높습니다.

---

## 2. 인터럽트 처리 과정 (Interrupt Handling)

인터럽트 신호가 들어오면 CPU와 OS는 다음과 같은 일련의 과정을 거쳐 이벤트를 처리합니다.

### 처리 루틴 흐름
1. **인터럽트 요청 수신**: 현재 명령어(Instruction) 실행을 마친 직후 인터럽트 라인을 확인합니다.
2. **컨텍스트 세이빙 (Context Saving)**: 
   - 현재 실행 중인 프로세스의 상태(PC, 레지스터 값, 모드 비트 등)를 해당 프로세스의 **PCB(Process Control Block)** 또는 **커널 스택(Kernel Stack)** 에 저장합니다.
   - 이는 나중에 중단된 시점부터 작업을 재개하기 위함입니다.
3. **인터럽트 벡터 테이블(IVT) 참조**:
   - 인터럽트 번호에 대응하는 **ISR(Interrupt Service Routine)** 의 시작 주소를 테이블에서 찾아 이동합니다.
4. **ISR 실행**: 실제 인터럽트를 처리하는 커널 코드(인터럽트 핸들러)를 수행합니다.
5. **복구 및 귀환**: 
   - 저장이 완료된 컨텍스트를 복원하고, `IRET`(Interrupt Return) 명령을 통해 원래 수행하던 작업으로 돌아갑니다.

## 3. 이중 동작 모드 (Dual-mode Operation)

시스템의 보안과 안정성을 위해 운영체제는 CPU 실행 권한을 두 가지 모드로 분리하여 관리합니다.

| 모드 | Mode Bit | 설명 |
| :--- | :---: | :--- |
| **유저 모드 (User Mode)** | 1 | 일반 사용자 프로그램이 실행되는 모드. 하드웨어 접근 및 특권 명령 실행이 제한됨. |
| **커널 모드 (Kernel Mode)** | 0 | 운영체제 커널이 실행되는 모드. 모든 하드웨어 자원과 명령어를 실행할 수 있는 특권 권한(Privileged Mode). |

- **특권 명령(Privileged Instruction)**: I/O 장치 제어, 타이머 설정, 모드 비트 변경 등 시스템 전체에 영향을 주는 명령어입니다. 오직 커널 모드에서만 실행 가능합니다.
- **모드 전환**: 유저 모드에서 시스템 콜이나 인터럽트가 발생하면 하드웨어적으로 Mode Bit가 0으로 바뀌며 커널 모드로 진입합니다.

---

## 4. 시스템 콜 (System Call)

사용자 프로그램이 운영체제의 서비스(파일 입출력, 프로세스 생성, 네트워크 통신 등)를 이용하기 위해 호출하는 유일한 인터페이스입니다.

### 작동 메커니즘
1. 사용자 프로그램이 특정 레지스터에 **시스템 콜 번호**를 세팅합니다.
2. 소프트웨어 인터럽트인 `trap` 명령을 호출합니다.
3. CPU가 커널 모드로 전환되며 제어권이 커널의 **시스템 콜 핸들러**로 넘어갑니다.
4. 커널은 **시스템 콜 테이블**을 참조하여 해당 번호에 맞는 커널 함수를 실행합니다.
5. 작업 완료 후 다시 유저 모드로 전환하여 결과를 프로그램에 반환합니다.

### 매개변수 전달 방식 (Parameter Passing)
사용자 영역의 데이터를 커널로 전달하는 세 가지 주요 방식입니다.
- **레지스터(Register)**: CPU 레지스터에 직접 값을 담아 전달. 가장 빠르지만 전달 가능한 데이터 크기가 제한적입니다.
- **메모리 블록 (Block/Table)**: 데이터가 담긴 메모리 주소만 레지스터를 통해 전달. 대량의 데이터를 전달할 때 가장 일반적인 방식입니다.
- **스택 (Stack)**: 데이터를 스택 영역에 Push하여 전달합니다.

---

> [!TIP] 시스템 콜은 왜 함수 호출(Function Call)이 아닌 인터럽트 방식을 사용하나요?
>
> 일반적인 함수 호출은 동일한 권한(유저 모드) 내에서 메모리 주소만 이동하지만, 시스템 콜은 **보호된 자원**에 접근해야 하므로 **권한 상승(User -> Kernel)** 과정이 반드시 필요하기 때문입니다.
