# CPU 스케줄링 (CPU Scheduling)

> **Q:** '선점(Preemptive)'과 '비선점(Non-preemptive)' 스케줄링의 결정적인 차이는 무엇인가?

> **A:** 주체인 **운영체제(OS)**가 실행 중인 프로세스로부터 CPU 주도권을 강제로 뺏을 수 있는지의 여부이다. 뺏을 수 있다면 선점형, 프로세스가 스스로 CPU(혹은 I/O 작업 등으로)를 반납할 때까지 온전히 기다려야 한다면 비선점형이다.
> 
> *참고: 현대 운영체제의 멀티태스킹은 주로 선점형으로 구현되며, Single-core 환경에서는 타이머 인터럽트를 통해 시분할(Time-sharing)로 동시성을 모사하고, Multi-core 환경에서는 여러 코어가 물리적으로 병렬성(Parallelism)을 확보하여 동시에 프로세스를 스케줄링한다.*

> **Q:** 효율성만 따지는 스케줄링에서 발생하는 '기아 현상(Starvation)'이란 무엇이며, 어떻게 해결하는가?

> **A:** 우선순위 기반 스케줄링이나 SJF에서 실행 시간이 긴(또는 우선순위가 낮은) 프로세스가 우선순위가 높은 프로세스들에게 계속 밀려 CPU를 영영 할당받지 못해 무한정 대기하는 상태를 말한다. 이를 해결하기 위해 대기 시간에 비례하여 특정 프로세스의 우선순위를 점진적으로 높여주는 **에이징(Aging)** 기법을 사용한다.

> **Q:** 라운드 로빈(RR) 방식에서 '할당 시간(Time Quantum)' 설정이 시스템 성능에 어떤 영향을 주는가?

> **A:** 할당 시간이 너무 길면 **FCFS(먼저 온 순서대로 처리)** 와 다를 바 없어 프로세스들의 응답성이 현저하게 떨어진다. 반대로 너무 짧으면 프로세스를 교체하는 **문맥 교환(Context Switch)** 비용(PCB 복원, 레지스터 상태 교체 등 로우레벨 구조 오버헤드)이 기하급수적으로 커져서, 실제 작업 처리보다 시스템 관리 업무에 CPU를 더 많이 쓰게 되어 전체 시스템의 처리량(Throughput)이 줄어든다.

> **Q:** 다단계 피드백 큐(MLFQ)는 왜 대화형 프로세스에 우선권을 주나요?

> **A:** 마우스 클릭이나 키보드 입력처럼 사용자와 직접 소통하는 I/O 위주(I/O Bound) 프로세스를 먼저 처리해야 사용자가 "컴퓨터가 빠르다"고 느끼는 **체감 성능(응답성)** 이 극대화되기 때문이다. MLFQ는 CPU를 짧게 쓰는 프로세스(대부분 대화형)에게 높은 우선순위를 유지시켜 이들의 대기 및 응답 시간을 줄여준다.
