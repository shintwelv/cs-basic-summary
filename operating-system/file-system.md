# 파일 시스템 (File System)

운영체제는 물리적인 저장 장치(HDD, SSD)를 사용자가 편리하게 사용할 수 있도록 **추상화(Abstraction)** 하여 관리한다.

## 1. 파일 시스템의 존재 이유

- 물리적 한계 극복: 사용자가 수조 개의 0과 1이 담긴 '블록 번호'를 직접 기억할 수 없으므로, 이를 '파일'이라는 논리적 단위로 관리한다.
- 추상화: 복잡한 하드웨어의 세부 사항을 숨기고 이름, 경로, 권한 등의 편리한 인터페이스를 제공한다.

## 2. 파일의 구조와 아이노드 (Inode)

- 메타데이터(Metadata): 파일의 이름(일부 시스템 제외), 크기, 생성 시간, 소유자, 접근 권한 등 파일 자체의 정보를 의미한다.

- 아이노드(Inode): 유닉스 계열 파일 시스템에서 파일 하나당 부여되는 '종합 정보 카드'이다.

- 직접 포인터: 작은 파일의 데이터를 즉시 가리켜 속도가 빠르다.

- 간접 포인터 (Single/Double/Triple): 큰 파일 수용을 위해 '색인 블록'을 계층적으로 연결한다. (속도와 용량 사이의 타협점)

## 3. 디렉터리 (Directory)

- 정체: 디렉터리 또한 하나의 '파일'이다.

- 역할: "사용자가 인식하는 파일 이름"과 "시스템이 인식하는 아이노드 번호"를 연결하는 매핑 테이블(Mapping Table) 역할을 수행한다.

- 루트 디렉터리: 모든 탐색의 시작점으로, 시스템에서 미리 약속된 특정 아이노드 번호(예: 2번)를 가진다.

## 4. 공간 관리 (Space Management)

- 불연속 할당: 파일을 여러 조각으로 나누어 빈 공간 어디든 저장하여 디스크 낭비를 줄인다.

- 비트맵(Bitmap): 각 블록의 사용 여부를 0과 1로 표시한 지도이다.

    - 장점: CPU의 비트 연산이 가능하여 연속된 빈 공간을 찾는 데 매우 빠르다.

    - 최적화: 지도가 너무 커지는 것을 방지하기 위해 구역(Chunk)별 요약 정보를 활용한다.

## 5. 신뢰성과 저널링 (Journaling)

- 문제: 정전 등으로 인해 데이터와 메타데이터 중 하나만 기록될 경우 데이터 불일치(Inconsistency)가 발생한다.

- 저널링: 실제 작업을 수행하기 전, **'작업 일지(Journal)'** 에 계획을 먼저 기록한다.

- 사고 발생 시 일지를 확인하여 완료되지 않은 작업은 취소(Undo)하거나 재실행(Redo)하여 시스템을 복구한다.

- 모드 선택: 속도가 중요하다면 메타데이터 저널링을, 안전성이 최우선이라면 전체 저널링을 선택한다.
