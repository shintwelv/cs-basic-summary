# 해시 테이블 (Hash Table)

## 기본 개념 및 성능

정의: 키(Key)를 해시 함수에 넣어 얻은 인덱스(Index)에 값(Value)을 저장하는 자료구조이다.

시간 복잡도:

- 평균: O(1) — 계산 한 번으로 데이터에 바로 접근한다.
- 최악: O(n) — 모든 데이터가 한 곳에서 충돌할 때 발생한다.

## 해시 충돌(Collision) 해결책

- Chaining: 같은 인덱스에 데이터가 들어오면 연결 리스트로 뒤에 이어 붙인다.
- Open Addressing: 충돌 시 배열 내의 다른 빈 공간을 찾는다.
- 선형 탐사(Linear Probing): 바로 다음 칸($+1$,$+2$...) 확인. (데이터가 뭉치는 '클러스터링' 위험이 있음)
- 이차 탐사(Quadratic Probing): 제곱수($+1^2$, $+2^2$, $+3^2$...)만큼 건너뛰며 확인.
- 이중 해시(Double Hashing): 두 번째 해시 함수로 건너뛸 보폭을 결정.

## 해시 함수 설계의 지혜

- 나눗셈법: index = `key % array_size`. 이때 `array_size`를 **소수(Prime Number)** 로 정해야 데이터가 골고루 퍼진다.
- 비트 연산 트릭: 컴퓨터가 빠른 & 연산을 쓰기 위해 배열 크기를 $2^n$으로 잡되, 하위 비트만 쓰이는 문제를 막기 위해 비트 뒤섞기(Mixing) 과정을 거친다.
- 문자열 해싱: 각 문자에 자릿수만큼 특정 값을 곱해 더하는 다항식 해시를 사용하며, 효율적인 계산을 위해 호너의 방법을 쓴다.

## 구조적 최적화 (Resizing)

- 부하율(Load Factor): 데이터가 약 70~80% 차면 성능 유지를 위해 배열 크기를 키운다.
- 재해싱(Rehashing): 배열 크기가 변하면 해시 결과값도 변하므로, 기존 데이터를 모두 새 위치로 다시 옮겨야 한다.
- 점진적 재해싱: 한꺼번에 옮길 때의 멈춤 현상을 막기 위해 데이터를 조금씩 나누어 이사시킨다.
